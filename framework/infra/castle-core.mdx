`Volo.Abp.Castle.Core` is the small package that lets ABP's interception system speak [Castle.Core](https://github.com/castleproject/Core) DynamicProxy. It is referenced by `Volo.Abp.Autofac` because Autofac is the container that actually builds the proxies.

## Module

```csharp framework/src/Volo.Abp.Castle.Core/Volo/Abp/Castle/AbpCastleCoreModule.cs
public class AbpCastleCoreModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddTransient(typeof(AbpAsyncDeterminationInterceptor<>));
    }
}
```

`AbpAsyncDeterminationInterceptor<TInterceptor>` wraps a user `IAbpInterceptor` so that DynamicProxy's sync invocation pipeline can dispatch into async code without deadlocks.

## Adapter types

| Adapter | Purpose |
| --- | --- |
| `CastleAsyncAbpInterceptorAdapter` | Adapts an `IAbpInterceptor` to Castle's `IInterceptor` interface; relies on `AbpAsyncDeterminationInterceptor` to detect async vs sync methods. |
| `CastleAbpMethodInvocationAdapter` | Wraps an `IInvocation` (Castle) as an `IAbpMethodInvocation` (ABP) for `void` / `Task` methods. |
| `CastleAbpMethodInvocationAdapterWithReturnValue` | Same as above for methods that return a value, propagating `ReturnValue` correctly. |
| `CastleAbpMethodInvocationAdapterBase` | Common base; centralizes argument capture and `MethodInfo` exposure. |

All four classes live under `framework/src/Volo.Abp.Castle.Core/Volo/Abp/Castle/DynamicProxy/`. You normally do not reference them directly — `Volo.Abp.Autofac` consumes them when registering services that ABP has flagged for interception.
