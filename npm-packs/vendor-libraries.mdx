The bulk of `npm/packs/` is a long list of tiny meta-packages that each wrap a
single third-party browser library and expose it through an `@abp/*` name and
an `abp.resourcemapping.js` mapping. The pattern lets a UI project pin
**framework-versioned** asset packages instead of upstream npm versions, and
gives ABP one place to upgrade them.

Each row below was generated from
`npm/packs/<dir>/package.json` — `dependencies` and `version: 10.5.0-rc.4`.

| `aspnetcore.mvc.ui` | only `ansi-colors` | Placeholder for MVC UI consumers — see [MVC UI bundles](/npm-packs/aspnetcore-mvc-ui). |

## Notes & conventions

- Every wrapper is at framework version **`10.5.0-rc.4`** at this snapshot. The
  `npm/packs/npm-check-updates.ps1` script (sibling to the package
  directories) is what the release process uses to bulk-update upstream
  versions across all packs.
- A package's `dependencies` is the entire contract — none of these directories
  contain JavaScript source beyond `abp.resourcemapping.js`. The mapping
  pattern is invariant:

  ```javascript
  module.exports = {
    mappings: { "@node_modules/<lib>/dist/*": "@libs/<dest>/" }
  };
  ```

- The CLI side that consumes these mappings is
  `framework/src/Volo.Abp.Cli.Core/Volo/Abp/Cli/LIbs/InstallLibsService.cs`.
