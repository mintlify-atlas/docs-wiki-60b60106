The Account module sits on top of [Identity](/modules/identity) and supplies the **end-user surface**: register, login, logout, forgot/reset password, account management, two-factor, external login, and the profile page. It also ships two host packages that turn a web app into a full OAuth2/OIDC authorization server — one for [OpenIddict](/modules/openiddict) and one for legacy [IdentityServer](/modules/identity-server).

## Application services

All under `modules/account/src/Volo.Abp.Account.Application/Volo/Abp/Account/`:

### `AccountAppService` (`AccountAppService.cs`)

Anonymous-friendly entry points used by the login/register flow:

```csharp
public virtual async Task<IdentityUserDto> RegisterAsync(RegisterDto input)
{
    await CheckSelfRegistrationAsync();
    await IdentityOptions.SetAsync();

    var user = new IdentityUser(GuidGenerator.Create(), input.UserName, input.EmailAddress, CurrentTenant.Id);
    input.MapExtraPropertiesTo(user);

    (await UserManager.CreateAsync(user, input.Password)).CheckErrors();
    await UserManager.SetEmailAsync(user, input.EmailAddress);
    await UserManager.AddDefaultRolesAsync(user);

    return ObjectMapper.Map<IdentityUser, IdentityUserDto>(user);
}
```

Other operations:

- `SendPasswordResetCodeAsync` → `IAccountEmailer.SendPasswordResetLinkAsync`.
- `VerifyPasswordResetTokenAsync` / `ResetPasswordAsync` → `UserManager.VerifyUserTokenAsync` and `ResetPasswordAsync`.
- `CheckSelfRegistrationAsync` honors the `Abp.Account.IsSelfRegistrationEnabled` setting.

### `ProfileAppService` (`ProfileAppService.cs`)

`[Authorize]` service for the logged-in user's own profile — get/update profile, change password, send/verify email/phone confirmation codes. Username updates are gated on `IdentitySettingNames.User.IsUserNameUpdateEnabled`.

### `DynamicClaimsAppService` (`DynamicClaimsAppService.cs`)

Single endpoint, `RefreshAsync`, that clears the user's entry in `IdentityDynamicClaimsPrincipalContributorCache` and rebuilds the `ClaimsPrincipal`. UIs call it after a permission/role change so the next request reflects the new claims without forcing a re-login.

## Emailing

`modules/account/src/Volo.Abp.Account.Application/Volo/Abp/Account/Emailing/`:

- `IAccountEmailer` / `AccountEmailer` use ABP's `IEmailSender` + `ITemplateRenderer` to send password-reset, email-confirmation and link-creation emails.
- `Templates/` ships default Razor email templates registered with the `ITemplateDefinitionProvider`.
- `AppUrlProviderAccountExtensions.cs` builds back-links using the host's configured `IAppUrlProvider` (so the email's "Reset password" button points to the correct frontend).

## Settings

`Settings/AccountSettingDefinitionProvider.cs` registers the two Account-specific settings: `Abp.Account.IsSelfRegistrationEnabled` and `Abp.Account.EnableLocalLogin` (constants on `AccountSettingNames`). They are read at runtime via `ISettingProvider` and configurable per tenant through [Setting Management](/modules/setting-management).

## Web hosts: OpenIddict vs IdentityServer

The base `Volo.Abp.Account.Web` package contains the generic Razor Pages under `Pages/Account/` — `Login.cshtml`, `Register.cshtml`, `ForgotPassword.cshtml`, `ResetPassword.cshtml`, `ResetPasswordConfirmation.cshtml`, `Manage.cshtml`, `Logout.cshtml`, `LoggedOut.cshtml`, `AccessDenied.cshtml`, `PasswordResetLinkSent.cshtml`. Two extension packages plug it into an OAuth2/OIDC pipeline:

- **`Volo.Abp.Account.Web.OpenIddict`** — adds `OpenIddictSupportedLoginModel.cs` so the `Login` page interprets the OpenIddict `returnUrl` (authorize request) and issues the correct sign-in. Recommended for new applications.
- **`Volo.Abp.Account.Web.IdentityServer`** — adds a `Consent.cshtml` page and IdentityServer4/Duende-aware login model. Use when the host runs the legacy [IdentityServer](/modules/identity-server) module.

Both packages reuse the same `AccountAppService` underneath, so business logic, emails, and settings are identical regardless of authorization-server choice.
