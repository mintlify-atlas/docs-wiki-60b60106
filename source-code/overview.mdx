Most ABP modules ship as compiled NuGet packages — you add the `Volo.Abp.Identity.Web` package and reference the binary. The repository also publishes a parallel **source-code** distribution of the same modules, where the module's source tree is shipped as a ZIP inside a NuGet content package. Two folders carry these packages:

- `source-code/` — the canonical set used by `abp add-module --with-source-code` for the open-source ABP modules.
- `studio/source-codes/` — the identical set mirrored for ABP Studio's source-code import flow.

## How a SourceCode package is built

Each project is a tiny `Microsoft.NET.Sdk` library that packs a single `*.zip` as `content`:

```xml source-code/Volo.Abp.Identity.SourceCode/Volo.Abp.Identity.SourceCode.csproj
<Project Sdk="Microsoft.NET.Sdk">
  <Import Project="..\..\configureawait.props" />
  <Import Project="..\..\common.props" />
  <PropertyGroup>
    <TargetFramework>net6.0</TargetFramework>
  </PropertyGroup>
  <ItemGroup>
    <None Remove="Volo.Abp.Identity.SourceCode.zip" />
    <Content Include="Volo.Abp.Identity.SourceCode.zip">
      <Pack>true</Pack>
      <PackagePath>content\</PackagePath>
    </Content>
  </ItemGroup>
</Project>
```

The ZIP next to it (e.g. `Volo.Abp.Identity.SourceCode.zip`) holds the full source tree of that module — `*.Domain`, `*.Application`, `*.HttpApi`, `*.Web`, etc. When a user runs `abp add-module Volo.Abp.Identity --with-source-code`, the CLI downloads this NuGet, extracts the embedded ZIP into the user's solution, and replaces the `PackageReference`s with `ProjectReference`s — implemented by `SolutionModuleAdder` + `SourceCodeDownloadService` (see [Project modification](/tooling/project-modification)).

`SourceCodes.slnx` aggregates every `*.SourceCode.csproj` so the build pipeline can produce all of them in one `dotnet pack`.

`studio/source-codes/` mirrors the same eighteen project folders one-for-one — every entry above exists at both `source-code/<name>` and `studio/source-codes/<name>`. The Studio copy is what ABP Studio reads when offering "Import as source code" from the module management UI.

## Why this exists

Source-code distribution is heavily used in two scenarios:

1. **Customisation.** The team wants to fork an official module's UI or business logic without losing the ability to merge upstream changes. Pulling the source into their solution makes the customisation a normal C# edit.
2. **Commercial / paid modules.** ABP Commercial modules typically ship in both binary and source-code form so license holders can audit or extend them. The `source-code/` folder in this repository is the open-source-only subset; commercial modules follow the same packaging pattern in the abpframework/abp-commercial repository.

`abp get-source <module-name>` (`GetSourceCommand.cs`) is the direct way to download just the source tree without adding the module to a solution. `abp add-module <module-name> --with-source-code` is the more common path — it downloads the same package, extracts it, and re-wires the solution to use the local projects via `NugetPackageToLocalReferenceConverter`.
